# دستور A.Z Hub v2 — النسخة العربية الموجزة

> مرجع سريع للمطوّرين ولوكلاء الذكاء الاصطناعي (Cursor / Claude / GPT).
> المرجع الكامل والملزم هو **[`AZHUB_V2_CONSTITUTION.md`](./AZHUB_V2_CONSTITUTION.md)** — عند أي تعارض فالإنجليزي هو المعتمد.

---

## 1) هوية التطبيق (لا تتغيّر)

- **مالك واحد فقط**، يحدّد بـ `OWNER_EMAIL`.
- **محفظة شخصية** للصكوك والاستثمارات ذات الدخل الثابت.
- ليست منصة تداول، ليست SaaS، ليست أداة محاسبة عامة، وليست شبكة اجتماعية.
- العملة **SAR** فقط، والدقة لخانتين (هللة).
- اللغات: عربي/إنجليزي، مع دعم RTL.
- الأولويات بالترتيب: **صحة مالية → قابلية تدقيق → وضوح → سرعة إدخال**.

---

## 2) قواعد مالية لا يجوز كسرها (FIN)

- **مصدر الحقيقة الوحيد** للأرقام المجمّعة هو `lib/finance/metrics.ts` عبر `GET /api/dashboard/metrics`.  
  الواجهة **تعرض فقط**، لا تجمع ولا تحسب.
- **كل حسابات المال** تمرّ عبر `lib/finance/money.ts` (`roundToMoney`, `sumMoney`, …).  
  ممنوع `toFixed` وممنوع الحساب بأرقام عائمة مباشرة.
- **المقاييس التسعة** في لوحة القيادة **مجمّدة**: الكاش، رأس المال النشط، NAV، Cash Drag، الأرباح المحققة، التدفقات 30/60/90، WAM، Default Rate، Active Annual Yield.  
  لا يُضاف/يُعاد تسمية/يُعاد تعريف أي منها دون موافقة المالك.
- **دفتر الكاش (`cash_transactions`) مزدوج القيد وملحَق فقط**:
  - deposit / cashflow_receipt → موجب
  - withdrawal / investment_funding → سالب
  - ممنوع إسقاط أي CHECK constraint.
- **استلام دفعة** (`PATCH /api/cashflows/:id/receive`) **ذرّي**: تحديث الحالة + إدخال قيد ليدجر في نفس الترانزاكشن.
- **حالة الاستثمار لا تُخزَّن** أبداً — تُشتق من `investment_status_view` (late/defaulted حد 90 يوماً).
- الجداول المخصّصة للتوزيعات المخصصة **لا تعود**؛ استخدم `cashflows.is_custom_schedule`.
- **Realized يعني Realized** — ممنوع أي fallback يخفي الواقع.

---

## 3) قواعد قاعدة البيانات (DB)

- `db/schema.ts` هو المصدر، وكل تغيير يحتاج **ملف migration** في `db/migrations/`.
- **ممنوع** `drizzle-kit push` على staging/production (يسقط كائنات SQL مثل الـ view).  
  يوجد سكربت محلي فقط اسمه `db:push:unsafe` للتجربة المحلية.
- كائنات SQL الخاصة (views/policies/triggers) تعيش في `db/sql/` وتُطبَّق عبر `db/migrate.ts`.
- Migrations **append-only**؛ لا يعاد تعديل ملف بعد الدمج.
- أي تغيير هدّام (DROP, تضييق نوع، إلخ) يحتاج **snapshot أولاً** عبر `portfolio_snapshots`.
- ممنوع جدول موازٍ يحمل نفس معنى جدول قائم.

---

## 4) الهوية البصرية والواجهة (DESIGN / UX)

- استخدم فقط مكوّنات `components/ui/*` (shadcn/ui) + ثيم Tailwind الحالي.
- ممنوع إضافة مكتبة UI ثانية، ثيم ثاني، أو مكوّن مكرَّر.
- حافظ على القالب: `AppSidebar` + `AppTopbar` + `<main className="container …">` (انظر `app/(app)/layout.tsx`).
- ممنوع إضافة عنصر Nav جديد في القائمة الجانبية دون موافقة.
- كل صفحة تعمل بسلاسة على شاشة **360 بكسل**، وأهداف اللمس ≥ 40 بكسل.
- لا **toast storm** (أكثر من إشعار لعملية واحدة).
- رسائل الخطأ **محدّدة ومفيدة** (قل أي حقل، أي صف). ممنوع "حدث خطأ ما".
- معالج الاستثمار (`investment-wizard.tsx`) **خطي** مع توجل بين **Duration ↔ End Date** ومعاينة عبر `/api/investments/preview-schedule` — لا يُحوَّل إلى تبويبات.

## 5) المصادقة والأمن (AUTH)

- **دخول المالك فقط** عبر Supabase باستخدام **كلمة مرور** لـ `OWNER_EMAIL`، ويجوز أن تُعرض في الواجهة كـ **PIN رقمي** مع بقاء المصادقة الداخلية على Supabase.
- كل API محمي يستدعي `requireOwner()`. كل صفحة محمية تستدعي `getOwnerSession()`.
- روابط المشاركة `share-links` هي الطريقة العامة الوحيدة، والإدخال فيها يبقى `needsReview=true` ولا يدخل في المقاييس حتى يعتمده المالك.
- ممنوع طريق مصادقة ثانٍ (Magic Link / OAuth / بصمة / Passkey / تخزين PIN محلي مستقل) دون تعديل صريح للدستور.
- `DATABASE_URL` **سيرفر فقط**. لا يظهر أبداً في ملف `"use client"`.

---

## 6) هندسة وبرمجة (ENG)

- منطق الأعمال على السيرفر (`app/api/*` و `lib/finance/*`).  
  مكوّنات العميل للعرض والإدخال فقط.
- استخدم **React Query** لحالة السيرفر — لا Redux/Zustand/Jotai موازية.
- كل بودي API يُفحص بـ **Zod** (عادة عبر `drizzle-zod`)، ورفض غير المطابق بـ 400.
- ممنوع ملفات موازية بنفس المسؤولية.
- PR يعدّل **> 10 ملفات** لفيتشر واحد **يُرفض** إلا بموافقة مسبقة.
- `legacy_reference/` **أرشيف قراءة فقط**؛ ممنوع الاستيراد منه.
- ممنوع رفع إصدارات أو إضافة مكتبات كأثر جانبي لطلب آخر.
- قبل أي دمج: `npm run lint && npm run typecheck && npm run test` أخضر.

---

## 7) النشر (DEP)

- ترتيب إلزامي: **local → staging → production**.  
  النشر المباشر للإنتاج ممنوع.
- الـ migrations تُطبَّق عبر `npm run db:migrate` باستخدام `DATABASE_URL` البيئة المستهدفة.
- قبل أي migration هدّام على الإنتاج: **snapshot** لذلك اليوم.
- خطة الرجوع: revert نشر Vercel → migration عكسي (لا تعديل تاريخ) → استعادة snapshot إن لزم.
- متغيّرات Vercel تطابق مفاتيح `.env.example` بالضبط، و `/api/health` هو البوابة.

## 8) بروتوكول وكلاء الذكاء (AI-R)

قبل أي تخطيط أو إجابة أو تعديل، يلتزم الوكيل بما يلي:

- **اقرأ الدستور أولاً**، والأصل هو قراءة النص الكامل وليس الاكتفاء بالملخّص عند وجود عمل فعلي على المشروع.
- لا تعدّل على:  
  `lib/finance/*` · `db/schema.ts` · `db/migrations/*` · `db/sql/*` · `app/api/**` · `components/investment-wizard.tsx`  
  إلا بعد: (1) تحديد أرقام القواعد المتأثّرة، (2) طلب إذن المالك، (3) تقديم خطة اختبار.
- لا "تحسّن" كوداً خارج نطاق المهمة.
- لا تخترع فيتشر لم يُطلب.
- لا ترفع إصدارات حزم بشكل جانبي.
- عند التوقّف بسبب قاعدة: **قف واذكر رقم القاعدة**، ولا تلتف حولها بملفات موازية.
- عامل `legacy_reference/` كأرشيف — لا تستورد منه.

### فحص ترشيح الموديل

- **AI-R-011** قبل الرد على أي طلب متعلق بالمشروع، يصنّف الوكيل المهمة: Architecture / Planning، Debugging، Implementation، UI/UX، Database / Backend، DevOps.
- **AI-R-012** قبل أي تنفيذ، يرشّح الوكيل الموديل الأنسب وفق الخريطة المعتمدة في الدستور الإنجليزي.
- **AI-R-013** قبل أي خطوة أخرى، يخرج الوكيل:
  - `Recommended model for this task: [MODEL NAME]`
  - `Reason: [short explanation]`
- **AI-R-014** لا يفترض الوكيل اعتماد الموديل تلقائياً.
- **AI-R-015** يتوقّف الوكيل بعد الترشيح وينتظر تأكيد المستخدم الصريح قبل التنفيذ.
- **AI-R-016** لا يقدّم الوكيل تنفيذاً كاملاً أو تغييرات كود أو حلاً نهائياً قبل تأكيد المستخدم أو تجاوزه للترشيح.
- **AI-R-017** إذا لم يطلب المستخدم تعمقاً تقنياً، يجب أن يكون الشرح **بسيطاً ومباشراً** ويفهمه غير المبرمج.

## 9) لوحة سريعة — مسموح / ممنوع

| مسموح ✅ | ممنوع ❌ |
|---|---|
| قراءة المقاييس عبر `/api/dashboard/metrics` | حساب مقياس في مكوّن React |
| استخدام `components/ui/*` | إنشاء Dialog/Select/Card مخصَّص |
| إضافة عمود بـ migration مع SQL COMMENT | تعديل migration بعد الدمج |
| إضافة اختبار Vitest جديد | حذف اختبار 1.005 أو اختبارات الجدولة |
| استخدام React Query للكاش | إضافة Redux/Zustand/Jotai |
| إضافة تعليق SQL لتوثيق عمود | حذف CHECK constraint على الكاش |
| إطلاق تنبيه واحد لكل عملية | Toast storm |
| نشر staging قبل production | نشر مباشر للإنتاج |

## 10) تعديل الدستور

- أي تغيير على القواعد يحتاج PR يعدّل `AZHUB_V2_CONSTITUTION.md`، يرفع رقم الإصدار في §0، ويحتوي موافقة صريحة من المالك.
- أي تعديل على الدستور الإنجليزي يجب أن يرافقه تحديث عربي موجز في `AZHUB_V2_DUSTUR_AR.md`، بشكل مختصر وملائم للسياق، لا كترجمة حرفية سطرًا بسطر.
